  Section 1: Purpose (Lines 1–14)

  The document opens by naming the project — Muxara — and immediately framing it as a "build brief for Claude Code," meaning it was written as a prompt/specification designed to be handed to Claude
  Code as an implementation directive. This is an unusual and notable meta-quality: the plan is both a product specification and an AI prompt.

  The purpose section makes a critical distinction right away: the problem Muxara solves is not terminal management. It explicitly calls out that tools like tmux already handle the mechanics of
  running multiple terminal sessions. Instead, the problem is cognitive overload — the human cost of juggling many concurrent AI-assisted coding sessions. It enumerates five specific symptoms: too
  many sessions, unclear identity, unclear state, difficulty knowing which session needs attention, and the time cost of reconstructing context from raw terminal output.

  This framing is strategically important because it positions Muxara as a human-factors tool, not a technical infrastructure tool. The value proposition is not "run more sessions" but "understand the
   sessions you're already running." This is a meaningful product insight — most developer tools focus on capability (doing more), while Muxara focuses on comprehension (understanding more). The
  distinction also serves as a design guardrail: any feature that doesn't directly reduce cognitive burden is out of scope. This is a disciplined, problem-first opening that avoids the common trap of
  leading with technology choices.

  Section 2: Core Intent (Lines 16–29)

  This section crystallizes the purpose into a single design mandate: build a persistent, always-visible overview of Claude Code sessions with enough context for rapid orientation and session
  switching. It then reinforces the boundary: Muxara is not a replacement for Claude Code, tmux, or the terminal. It is a "thin orchestration and observability layer."

  The most valuable part of this section is the five questions the user should be able to answer at a glance. These function as acceptance criteria in disguise — if the shipped product doesn't let a
  user answer all five questions quickly, it has failed. The questions progress from basic inventory ("what sessions exist?") through identity ("which belongs to which workstream?") and status
  ("active, blocked, or ignorable?") to context ("what's the recent history?") and finally decision-support ("which should I jump into next?"). This is a well-structured cognitive hierarchy: awareness
   → identification → triage → context → action.

  The phrase "thin orchestration and observability layer" is doing heavy lifting. "Thin" constrains scope. "Orchestration" implies it can create and manage sessions, not just observe them.
  "Observability" borrows from the infrastructure/SRE world, implying that sessions are running systems whose internal state should be externalized and made inspectable. This is a sophisticated
  framing for what could otherwise be described as "a tmux GUI."

  Section 3: Product Vision (Lines 31–45)

  This section introduces the mission-control metaphor, which is the emotional and aesthetic north star for the product. Mission control implies calm authority, information density without chaos, and
  the ability to monitor many things simultaneously while focusing on the one that matters most.

  The desired UX is laid out in six bullet points: persistent window, compact card grid/list, cards with orientation and recency, click-to-jump, a plus button for new sessions, and graceful scaling.
  These are concrete enough to design from but abstract enough to allow creative interpretation.
